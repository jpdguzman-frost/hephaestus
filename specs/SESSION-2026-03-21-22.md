## Key Architectural Learnings

### 1. Deterministic Pattern Application Failed
**What happened**: We built automatic pattern overrides in handlePluginTool — when create_node was called with a somRole, patterns were fetched and applied mechanically.

**Why it failed**:
- **Role taxonomy too coarse**: `somRole: "value"` captures both CTA button labels (16px) and table data values (12px). A fontSize pattern from data values broke CTA labels.
- **Context lost in extraction**: A "card" in settings has different padding than a "promo card" in a success screen. Raw (role, property) → value strips the context.
- **Override logic can't be universally correct**: "Only fill undefined" means patterns never apply (Claude always sets values). "Always override" means wrong-context patterns break good values.
- **Cross-brand patterns mix incompatible conventions**: Different brands have different design languages at the property level.

**Resolution**: Removed automatic overrides entirely (commit `235bbbc`). Patterns inform Claude's decisions, they don't replace them.

### 2. Inference + Context > Deterministic Rules
**The insight**: Patterns should be **guides** that Claude reads and reasons about, not **rules** that are mechanically applied. Claude has the contextual understanding to know when a pattern applies and when it doesn't.

**The vision-driven refinement loop**:
```
1. BUILD    → Rex builds screen from reference
2. CAPTURE  → Claude screenshots via Figma MCP
3. EVALUATE → Claude reads style guide + looks at screenshot
4. REFINE   → Claude applies contextual changes via update_node
5. VALIDATE → Claude screenshots again to confirm
```

### 3. Style Guide > Property Patterns
**What works**: A natural language principles document that Claude reads before building:
- Typography: font preferences, weight system, size ranges
- Spacing: grid system, relationships between elements
- Radii: proportional to element size, not absolute values
- Colors: derive from source screen palette, not prescribed hex values

**What doesn't work**: Raw per-node patterns like `{role: "card", property: "paddingTop", value: 12}` — too specific, too context-dependent.

### 4. Template-Based Builds Work Best
**Observation**: When we rebuilt the acorns "Transfer Complete" screen directly from the refined template SOM, it looked great. The template preserves ALL context — structure, spacing, fonts, colors. No extraction needed.

**Cold builds from screenshots** are decent but inconsistent. The style guide helps but can't fix structural quality issues.

### 5. Colors Should Be Derived, Not Prescribed
User feedback: Colors come from the SOURCE screen being built (Osiris provides color palette data). The style guide provides derivation principles ("primary text = darkest from palette") not hex values.

### 6. Radii Should Be Relational, Not Absolute
User feedback: Instead of "cards = 12px", capture relationships — "buttons use pill (height/2), cards use moderate rounding, inputs match cards." Claude infers the right value for each context.

## Known Bugs & Issues

### 1. Rex Can't Update textStyle on Existing Text Nodes
**Severity**: High — blocks the refinement workflow
**Behavior**: `update_node` with textStyle changes (fontSize, fontFamily, fontWeight) silently fails. The command succeeds but values don't change in Figma.
**Impact**: Can't apply the Phoenix typeset (HeyMeow Rnd / BarkAda) programmatically. Font swaps must be done manually.
**Location**: Likely in `plugin/executors/nodes.ts` → `applyProperties()` text handling
**Fix needed**: The text property update path needs to load fonts before applying changes, similar to how `executeSetText` handles it.

### 2. Rex Can't Load Custom Fonts (HeyMeow Rnd)
**Severity**: Medium — affects Project Phoenix specifically
**Behavior**: Setting fontFamily to "HeyMeow Rnd" or "BarkAda" falls back to Inter
**Root cause**: Custom fonts need to be pre-loaded in `plugin/fonts.ts` or loaded on-demand via Figma's font resolution
**Workaround**: User manually swaps font families after Rex sets structure

### 3. Plugin Chat History Truncated
**Severity**: Low — cosmetic
**Behavior**: Plugin chat UI only renders the last few messages, not full history. "Previous messages" link exists but older messages don't render inline.
**Location**: `plugin/ui.html` chat rendering

### 4. Old Osiris MCP Tools Still Registered
**Severity**: Low — dead code
**Tools**: `capture_delta`, `list_deltas`, `extract_principles`, `get_principles`, `get_refinement_context`
**These reference removed API endpoints from the old refinement layer approach. Should be cleaned up from `mcp-server/index.js`.
