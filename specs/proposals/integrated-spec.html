Template Scoring Algorithm

find_template uses a weighted scoring model to rank matches. Only templates where supersededBy is null (head of chain) are returned.

Signal Weight Logic
Screen type exact match 0.35 Exact = 1.0, partial (shared prefix) = 0.6
Tag overlap 0.20 Jaccard similarity
Mood match 0.15 Exact = 1.0, compatible = 0.5
Recency 0.15 1 - (daysSinceRefinement / 180), clamped [0, 1]
Usage count 0.10 min(usageCount / 20, 1.0)
Generation 0.05 min(generation / 5, 1.0)

Part 5 — Data Flow

Build → Refine → Learn → Reuse

The complete learning cycle. Every refinement session makes the system better. Templates capture whole-screen decisions, patterns capture cross-screen preferences.

01
Build

Search for a Template, Fall Back to Reference

Claude calls osiris_find_template with brand, screen type, tags, and mood. If a template matches (score > 0.3), Claude uses its SOM as the build plan, populating content slots and replacing brand-specific properties. If no template, Claude falls back to osiris_search_screens + osiris_get_screen_som and applies any confirmed property patterns (confidence >= 0.7). Rex builds via create_node with BuildManifest tracking every node-to-role mapping.

osiris_find_template osiris_get_template osiris_search_screens osiris_get_screen_som rex create_node
02
Refine

Designer Edits, Plugin Observes Silently

After build completes, the plugin starts an observation window. The documentchange listener captures every property change within the built frame's subtree. Changes are buffered and flushed to the relay every 2 seconds via existing HTTP polling. The BuildManifest maps each changed Figma node ID back to its SOM role. Window closes on: idle 60s, designer says "done", new build starts, or page switch.

plugin/observer.ts documentchange BuildManifest.get() change buffer flush
03
Learn

Classify, Filter, Extract, Store

The refinement recorder assembles change buffers into RefinementRecords and stores them in MongoDB. The adversarial filter classifies each change into 6 categories: revert detection, artifact cleanup, AI error correction, platform convention, content-driven, or genuine preference. Only genuine preferences enter the pattern pipeline. Patterns are grouped by (brandId, somRole, property). Confidence scores accumulate: 5+ consistent observations reach "confirmed" status. 8+ auto-promote silently.

refinement/recorder.ts refinement/filter.ts refinement/patterns.ts refinement_records property_patterns
04
Reuse

Save Template, Apply Patterns Next Time

When the observation window closes, Claude prompts: "Save this as a template?" If yes, Claude runs extract_som on the refined frame, identifies content slots, tags property annotations (brand vs. taste), and calls osiris_save_golden_template. The template enters the supersession chain. Next time the same screen type is requested, the template is found first. On no-template builds, confirmed patterns auto-adjust node values. Override detection tracks whether the designer keeps or reverts pattern-applied values.

rex extract_som osiris_save_golden_template refinement/applicator.ts override detection

Cold Start: Zero Templates

The first build works identically to today: reference search, SOM extraction, Rex build. The designer refines heavily (15-20 changes). Claude saves the result as a Generation 1 template. After 5-10 sessions covering common screen types, most requests hit a template. No special cold-start infrastructure needed. Optional: batch import 10-20 existing Figma screens to seed templates in one session.
